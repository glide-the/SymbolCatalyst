# 角色设定评价系统接入说明

这份文档把外部需求 `角色设定评价系统设计师与引擎规范` 中与接入相关的内容，整理成当前 `SymbolCatalyst` 项目的上下文说明，目标是回答三个问题：

1. 这个项目里已经有哪些可复用资产。
2. 这些资产如何映射到“角色设定评价系统”的运行时概念。
3. 后续真正做接入时，转换层应该长成什么样。

## 1. 结论先行

`SymbolCatalyst` 最合理的定位不是“角色设定评价系统本体”，而是它的上游资产仓与协议说明层。

从外部需求看，核心工作并不是继续写更多概念稿，而是把 `game-engine-package` 的打包结果收束成一个稳定的、可被当前运行时读取的配置协议。对当前项目来说，已经具备的基础如下：

- 有产品级系统稿，定义了角色卡、世界观、规则卡、章节、现实映射与商业化结构。
- 有 `symbol_engine_20260218_v2.0.json` 这类结构化主数据。
- 有研究笔记、状态图与伪代码，能补足 JSON 里没有的语义说明。
- 有三个上游技能目录，说明这些文件是如何被生成出来的。

因此，接入的正确方向是：

1. 把当前项目视为“上游产物归档层”。
2. 由独立转换模块读取 `files/` 下的结构化数据与补充语义。
3. 输出角色设定评价系统的运行时配置，而不是直接消费原始包内容。

## 2. 外部需求提炼

结合外部文档，接入目标可以压缩为四条：

### 2.1 主事实源要明确

`symbol_engine*.json` 是主事实源，负责提供：

- 符号表
- 状态定义
- 规则命题
- 执行规则
- 任务编排
- 输出索引

### 2.2 Markdown 不是噪音，而是补充语义层

`research_notes` 里的 Markdown 不能被当成纯附件处理，因为其中包含：

- 命题边界与失败条件
- rewrite hooks
- 幻想结、权力链、追问种子
- 对 JSON Schema 的解释性约束

### 2.3 接入模块必须独立

外部需求已经明确要求：

- 转换逻辑应当是一个单独模块。
- 不直接破坏现有运行时编排。
- 通过大模型分析协议补足语义缺口，但关键字段仍要可追溯、可校验。

### 2.4 目标不是“展示文档”，而是“可读配置”

最终产物应该能被角色设定评价系统直接读取，用于：

- 角色选择后的初始化
- 剧情节点生成
- 决策触发与规则判定
- 现实映射变量更新
- 规则改写和记忆记录

## 5. 建议的转换协议

为保证运行时不直接依赖异构文件，建议在接入层产出一个统一配置对象。命名可以沿用外部设计中的思路，例如 `evaluation-pack.v1.json`。

建议至少包含以下块：

| 块 | 内容 |
| --- | --- |
| `meta` | 来源名称、版本、领域、生成时间 |
| `symbolDictionary` | 结构化符号表 |
| `variableModel` | 变量清单、默认值、映射权重 |
| `ruleCatalog` | 命题、触发条件、效果、边界、失败条件、rewrite hooks |
| `stateModel` | 当前状态、状态历史、状态转移 |
| `runtimeRuleCards` | 角色设定评价系统直接展示与执行的规则卡 |
| `promptAssets` | 系统提示词、约束、示例片段 |
| `resourceManifest` | 补充素材、图表、研究笔记和日志索引 |
| `trace` | sourcePath -> targetPath 的映射证据 |
| `reviewQueue` | 低置信字段、无法确定的 join、需要人工确认的项 |

## 6. 转换策略建议

### 6.1 先确定性映射，再让大模型补语义

适合确定性处理的内容：

- `engine_config`
- `metadata`
- `symbol_table.symbols`
- `symbol_table.variables`
- `state.current/history/transitions`
- `rules[].priority`
- 文件路径、哈希、资源清单

适合用大模型补的内容：

- 命题边界与失败条件的整理
- 规则卡可读文本的压缩与归一
- Markdown 中 JSON 块和自然语言解释的对齐
- 从研究笔记里提炼 prompt 片段

### 6.2 输入源优先级

建议用下面的优先顺序处理冲突：

1. `files/data/symbol_engine_20260218_v2.0.json`
2. `files/research_notes/01_data_structures.md`
3. `files/research_notes/02_rule_deck.md`
4. `files/research_notes/03_outputs_O1_O5.md`
5. `files/data/data_summary.md`
6. 顶层体验型文档（`02` 到 `06`）

原因很简单：越靠前的内容越结构化，越适合作为运行时协议依据；越靠后的内容越偏产品表达和体验包装。

### 6.3 必须保留 trace

外部文档反复强调“可追溯”，因此转换层不应该只输出结果，还要输出来源证据：

- 目标字段来自哪个文件
- 来自哪一段文本或哪个 JSON 字段
- 是否经过 LLM 改写
- 当前置信度是多少

否则后续一旦规则表现异常，无法知道问题出在源文件、映射规则还是 LLM 补全。

## 7. 当前项目已经具备什么，还缺什么

### 已具备

- 角色设定评价系统所需的大部分设计样本
- 一份结构化 `symbol_engine` JSON
- 状态流转和规则引擎伪代码
- 上游技能的说明目录
- 顶层产品文档和系统文档

### 仍缺失

- 真正的转换模块实现
- 最终打包产物，如 `game-engine-generator-prompt.md` 或 zip 包
- 针对角色设定评价系统数据结构的正式目标 Schema
- 自动化校验与回归样本

## 8. 对当前项目的建议定位

如果继续沿这个方向推进，`SymbolCatalyst` 最适合作为：

### 8.1 设计与协议仓

集中维护：

- 上游技能定义
- 结构化引擎样本
- 接入协议与样例
- 产品叙事与系统说明

### 8.2 不是运行时实现仓

不建议在这里直接长出完整的角色设定评价系统运行时代码，原因是职责会混乱：

- 这里的文件以说明、样本和技能资产为主。
- 运行时代码需要更明确的数据模型、测试和发布边界。
- 独立转换模块应更靠近真正消费配置的工程。

## 9. 下一步最小落地动作

如果后续要把这个说明推进成实现，建议按下面顺序落地：

1. 在运行时工程里冻结目标 Schema。
2. 以 `files/data/symbol_engine_20260218_v2.0.json` 为第一份 golden sample。
3. 补一个独立转换模块，按“确定性解析 -> LLM 补语义 -> Schema 校验 -> reviewQueue”四段走。
4. 把 `02_角色卡示例x8.md`、`04_规则卡示例x12.md`、`05_叙事章节Demo与动态改写算法.md` 当作运行时显示层样本，而不是主事实源。
5. 给每次转换输出 trace 和 compatibility report。

## 10. 一句话总结

`SymbolCatalyst` 已经具备“角色设定评价系统”接入所需的大部分上游资料，但它现在更像协议和资产仓。真正缺的不是更多概念文案，而是一层把 `symbol_engine + research_notes + prompt assets` 稳定收束成运行时配置的转换模块。
