# SymbolCatalyst

`SymbolCatalyst` 是一个以文档和资产为中心的工作区，用来整理上游引擎技能产物，并把它们收束成可接入“角色设定评价系统”的中间层说明。

这个项目当前不是运行时代码仓库，而是一个“设计蓝图 + 结构化样本 + 技能资产 + 接入说明”的组合包，解决的是两个问题：

1. 把 `symbol-engine-generator`、`object-engine-generator`、`game-engine-generator` 的输出放进同一个可追踪目录。
2. 为后续把这些产物接入角色卡叙事系统、规则改写系统、现实映射系统提供统一入口。

## 这个项目和“角色设定评价系统”的关系

外部需求把目标说得很明确：现有 `game-engine-package` 的打包结果，需要被转换成角色设定评价系统可读的配置，而不是直接把上游 JSON 和 Markdown 原样喂给运行时。

因此，`SymbolCatalyst` 在整条链路里的位置应当理解为：

1. 上游技能输出的归档层。
2. 运行时协议的分析层。
3. 未来转换模块的文档依据。

换句话说，真正的运行时系统负责：

- 角色选择与 Persona 初始化
- 场景生成与决策推进
- 规则触发与动态改写
- 现实映射变量结算
- 记忆、存档和终局判定

而 `SymbolCatalyst` 负责提供：

- 角色、规则、世界观和流程样本
- `symbol_engine` 结构化数据
- 上游技能的语义约束
- 如何把这些内容转成运行时配置的说明

## 当前边界

项目内已经存在：

- 一套完整的产品与系统设计文档
- 一份 `symbol_engine` 样本配置
- 对应的研究笔记、图表源文件和伪代码
- 三个上游技能目录

项目内当前没有看到的内容：

- 面向运行时的正式转换模块代码
- `game-engine-generator-prompt.md` 这类最终打包输出
- zip 交付包
- 真正消费这些产物的前端或服务端工程

因此，这个仓库更接近“设计与协议工作区”，不是最终产品仓库。
