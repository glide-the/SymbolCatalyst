# 角色设定评价系统设计师与引擎规范分析

这份文档把外部 Notion 目录 `角色设定评价系统设计师与引擎规范` 中的设计信息整理成 `SymbolCatalyst` 项目可直接复用的说明，目标是回答四个问题：

1. 外部规范真正要求交付什么。
2. 这些要求依赖哪些核心数据结构和运行时概念。
3. `SymbolCatalyst` 当前已经有哪些资产可以接住这些要求。
4. 本项目下一步应该把工作推进到哪一层，而不是误把自己当成运行时本体。

## 1. 输入来源

本次分析主要基于两个源文件：

| 来源 | 作用 |
| --- | --- |
| `角色设定评价系统设计师与引擎规范.md` | 给出接入目标、范围、转换模块设计、映射协议、测试与排期 |
| `系统设计师与引擎规范系统提示词.md` | 给出符号引擎的数据结构、公理、运行阶段、输出物和规则卡模板 |

可以把两者理解为：

- 第一份文档回答“要接什么、怎么接、做到什么算验收”。
- 第二份文档回答“上游符号引擎长什么样、输出什么、哪些结构必须保留”。

## 2. 外部规范提炼出的核心结论

### 2.1 目标不是直接读取包，而是做独立转换模块

外部规范反复强调，`game-engine-package` 不能直接作为角色设定评价系统的运行时输入，而应当经过一个单独的转换模块处理。这个模块需要：

- 读取 `symbol_engine*.json` 这类结构化规则资产。
- 读取 Markdown、Prompt、素材索引等补充语义文件。
- 输出角色设定评价系统当前可以消费的配置对象。
- 保留从源字段到目标字段的 trace，避免后续无法追责。

因此，`SymbolCatalyst` 更适合承担上游资产归档与协议说明角色，而不是直接变成运行时工程。

### 2.2 主事实源是 `symbol_engine`，Markdown 是语义补层

外部规范已经明确优先级：

1. `symbol_engine*.json` 是主事实源。
2. `data/*.md`、研究笔记、Prompt 说明是补充语义层。
3. 遇到不一致时，应优先信任结构化数据，再把自然语言内容放入 review 或 trace。

这意味着接入链路不能只做 Markdown 摘要，也不能只做 JSON 直通，而必须同时处理：

- 可确定解析的结构化字段。
- 需要补足边界、失败条件、rewrite hooks 的语义文本。

### 2.3 目标配置至少要落到运行时可读结构

外部设计把目标运行时描述得很清楚，至少要能覆盖：

- `RuleCard[]` 这类规则卡片结构。
- 三维属性或现实映射变量。
- 剧情推进、任务调度所需的 Prompt 片段。
- 初始化、规则判定、状态更新、回退兜底等运行时入口。

从接入层角度看，这要求转换结果不能只是“分析报告”，而必须是接近运行时 Schema 的配置包。

### 2.4 转换链路应该是“确定性解析 + LLM 语义映射 + 校验 + 回退”

外部方案最重要的工程结论是：

1. 先做确定性解析，处理元数据、状态、规则编号、路径清单等稳定字段。
2. 再让大模型处理边界、失败条件、规则卡文案、Prompt 片段等语义补全。
3. 对输出做 Schema 校验。
4. 校验失败时，把错误反馈给模型重试，或者降级到 review queue。

这个结论非常关键，因为它决定了本项目后续不该把所有解析都丢给模型，也不该把所有字段硬编码死。

## 3. 外部规范要求保留的关键结构

### 3.1 符号引擎核心实体

从“系统设计师与引擎规范系统提示词”中，可以提炼出上游引擎的最小结构集：

| 实体 | 含义 |
| --- | --- |
| `Symbol` | 抽象符号节点，带情绪向量、来源片段、载体信息 |
| `FantasyKnot` | 重复信息与幻想结，负责描述高频结构张力 |
| `IndexicalOperation` | 切片、凝固、对象化等范指动作 |
| `Proposition / Rule` | 命题规则，必须显式声明边界与失败条件 |
| `PowerChain` | 权力链运动，用于解释改写和规则迁移 |

这些结构不能在接入过程中被抹平，否则下游运行时会失去规则来源、改写路径和失败条件。

### 3.2 运行时循环

外部规范把系统运转模式拆成了稳定阶段：

1. 摄入话语流与工具信息。
2. 提取重复模式。
3. 做对象化和符号化。
4. 生成幻想结、命题、边界和失败条件。
5. 做追问驱动与巫术风险检测。
6. 基于权力链和现实映射变量做规则改写。
7. 输出规则集、改写日志、沉默报告和追问种子。

这说明接入层不应只交付静态卡牌，还应该尽量保留：

- 规则来源
- 改写痕迹
- 风险标记
- 可追问的问题种子

### 3.3 目标产物建议

结合外部设计文档中的映射协议，较合理的目标配置可以整理为：

| 配置块 | 内容 |
| --- | --- |
| `meta` | 来源名称、版本、领域、生成时间 |
| `symbolDictionary` | 符号表及其定义 |
| `variableModel` | 变量与现实映射权重 |
| `ruleCatalog` | 命题、触发条件、效果、边界、失败条件 |
| `runtimeRuleCards` | 运行时直接显示和执行的规则卡 |
| `stateModel` | 当前状态、历史、状态转移 |
| `promptAssets` | 系统提示词、约束、示例片段 |
| `resourceManifest` | 素材、图表、日志、研究笔记索引 |
| `trace` | 源字段到目标字段的证据链 |
| `reviewQueue` | 无法确定或低置信字段 |

## 4. 和 `SymbolCatalyst` 当前资产的映射

`SymbolCatalyst` 不是空仓，它已经有一套很适合作为接入前置层的资产。可以按下面方式理解：

| 当前项目资产 | 在接入链路中的角色 |
| --- | --- |
| `02_角色卡示例x8.md` | Persona/Card 结构样本 |
| `03_世界观骨架模板与示例.md` | 世界观和设定模板 |
| `04_规则卡示例x12.md` | 规则卡表达方式与运行时文案参考 |
| `05_叙事章节Demo与动态改写算法.md` | 场景推进和规则改写逻辑样本 |
| `files/data/symbol_engine_20260218_v2.0.json` | 结构化主输入 |
| `files/research_notes/*` | 语义补层和约束说明 |
| `files/charts/state_flow.mmd` | 状态流转关系 |
| `files/logs/rule_engine_pseudo.txt` | 伪代码与执行参考 |
| `skills/*` | 上游产物的生成来源说明 |

因此，对 `SymbolCatalyst` 最准确的定位是：

- 上游产物归档层
- 协议和映射说明层
- 接入前的结构化样本仓

而不是：

- 直接消费这些配置的前端运行时
- 真正执行 4-Task 编排的应用仓
- 已经完成转换模块实现的代码仓

## 5. 对当前项目的落地建议

### 5.1 本项目内应该继续沉淀什么

在 `SymbolCatalyst` 里，最值得继续补齐的是：

- 对外部规范的分析文档
- 对现有资产和目标运行时之间的映射说明
- 对未来转换模块输入输出协议的整理
- 对 trace、reviewQueue、兼容性策略的文档化约束

### 5.2 本项目内不必冒进实现什么

当前项目没必要直接承担以下职责：

- 角色设定评价系统的运行时 UI
- 完整的转换 CLI 与服务模块
- 4-Task 编排接入代码
- 面向最终玩家的交互逻辑

这些内容更适合放在真实消费配置的运行时项目里实现。

### 5.3 最小下一步

如果只做最小推进，建议分三步：

1. 在 `SymbolCatalyst` 里固定分析文档、映射表和目标配置草案。
2. 在消费方工程中实现独立转换模块，遵守“确定性解析 + LLM + 校验 + 回退”的链路。
3. 用现有 `symbol_engine_20260218_v2.0.json` 做第一份真实样本，验证 `RuleCard`、变量和 Prompt 片段能否完整落地。

## 6. 和现有顶层文档的关系

当前仓库顶层已有一份 `08_角色设定评价系统接入说明.md`，它更偏“面向当前项目的接入总结”。本文件则更偏“对外部 Notion 规范本身的结构化分析”。

两者的关系可以理解为：

- `doc/角色设定评价系统设计师与引擎规范分析.md`
  负责整理源规范里的设计信息和工程结论。
- `08_角色设定评价系统接入说明.md`
  负责把这些结论翻译成 `SymbolCatalyst` 当前项目语境下的接入判断。

## 7. 一句话结论

外部“角色设定评价系统设计师与引擎规范”真正要求的，不是再写一份概念稿，而是把上游 `game-engine-package` 产物通过独立转换模块收束成可校验、可追溯、可被运行时直接消费的配置；`SymbolCatalyst` 当前最适合承担的正是这个转换工作之前的资产归档与协议说明层。
