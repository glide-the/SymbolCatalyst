# 角色设定评价系统设计师与引擎规范分析

这份文档把外部 Notion 目录 `角色设定评价系统设计师与引擎规范` 中的设计信息整理成 `SymbolCatalyst` 项目可直接复用的说明，目标是回答四个问题：

1. 外部规范中描述的不同 skill 各自产出什么，它们如何组成完整的转换流水线。
2. 这条流水线的最终产物 `game-engine-package` 应满足什么结构要求才能被角色设定评价系统直接导入。
3. `SymbolCatalyst` 当前已经有哪些资产可以接住这些要求。
4. 本项目下一步应该把哪些 skill 集成到 agent 中，完成端到端的包生成与导入。

## 1. 输入来源

本次分析主要基于两个源文件：

| 来源 | 作用 |
| --- | --- |
| `角色设定评价系统设计师与引擎规范.md` | 给出三阶段 skill 流水线定义、接入目标、映射协议、测试与排期 |
| `系统设计师与引擎规范系统提示词.md` | 给出符号引擎的数据结构、公理、运行阶段、输出物和规则卡模板 |

可以把两者理解为：

- 第一份文档回答"skill 流水线怎么跑、产出什么、怎么打包、怎么导入"。
- 第二份文档回答"上游符号引擎长什么样、输出什么、哪些结构必须保留"。

## 2. 核心结论：skill 就是转换模块系统

### 2.1 三个 skill 本身就是转换流水线

外部规范中描述的三个 skill 不是"需要额外编写转换模块去消费的上游产物"，而是**它们本身就构成了完整的转换流水线**。每个 skill 是一个具体的产出阶段：

| 阶段 | Skill | 输入 | 产出 |
| --- | --- | --- | --- |
| 1 | `symbol-engine-generator-v2.0` | 话语流/交互数据 | 符号表、状态定义、规则命题 (O1-O5) |
| 2 | `object-engine-generator-v1.0` | 阶段 1 的符号配置 + 研究笔记 | 可执行引擎模板 (`symbol_engine*.json`) + Schema + 可视化 |
| 3 | `game-engine-generator-v1.0` | 阶段 2 的引擎配置 + 体验文档 | 系统提示词 + `game-engine-package` 压缩包 |

因此，`SymbolCatalyst` 的正确定位不只是"上游资产归档仓"，而是**承载这三个 skill 的 agent 工作空间**。把 skill 放入 agent 中，按照对应的流程执行，就可以输出 `game-engine-package` 压缩包。

### 2.2 `game-engine-package` 可直接导入角色设定评价系统

外部规范的关键结论是：`game-engine-package` 压缩包是角色设定评价系统的**直接导入物**，而不是需要再经过一层转换才能使用的中间产物。

这意味着：

- 压缩包内的结构必须符合角色设定评价系统的导入协议。
- 包内 `symbol_engine*.json` 是主事实源，承载动态游戏规则内容。
- 包内其他资源（Markdown、Prompt、素材索引）是游戏素材和场景 Prompt。
- 导入后，角色设定评价系统通过自身的读取逻辑消费这些配置，不需要外部转换模块。

### 2.3 压缩包内的资源路径必须是相对路径

这是一个关键的工程约束：`game-engine-package` 压缩包中的所有资源引用必须使用**相对路径**，而非绝对路径。原因是：

- 压缩包需要在不同环境中可移植。
- 角色设定评价系统在导入时，会将包解压到自己的工作目录下。
- 如果使用绝对路径，包在目标机器上会因为路径不存在而失败。

这要求 skill 3 (`game-engine-generator-v1.0`) 在打包阶段必须：

1. 将所有文件路径转换为相对于包根目录的路径。
2. 在 `symbol_engine*.json` 中的 `outputs.visualizations`、`outputs.logs`、`outputs.data_files` 等字段使用相对路径。
3. 在素材索引 (`index.json`) 中使用相对路径引用图片、字体等资源。

### 2.4 主事实源是 `symbol_engine`，Markdown 是语义补层

外部规范已经明确优先级：

1. `symbol_engine*.json` 是主事实源（动态游戏规则内容）。
2. `data/*.md`、研究笔记、Prompt 说明是补充语义层（游戏素材和场景 Prompt）。
3. 遇到不一致时，应优先信任结构化数据，再把自然语言内容放入 review 或 trace。

### 2.5 转换工作发生在 skill 流水线内部，而非外部模块

外部规范中提到的"转换函数"是指：skill 流水线在执行过程中，通过大模型分析转换协议，将自然语言输入逐步结构化为 `game-engine-package` 的格式。这个转换过程已经内化在三个 skill 的执行阶段中：

- Skill 1 负责语义提取与符号化（"确定性解析"部分）。
- Skill 2 负责 Schema 形式化与校验（"Schema 校验"部分）。
- Skill 3 负责交付打包与路径标准化（"LLM 语义映射 + 打包"部分）。

不需要再写一个独立的 `enginePackConverter` 模块。

## 3. Skill 流水线的产物结构

### 3.1 Skill 1 产出：符号引擎核心实体

从 `symbol-engine-generator-v2.0` 的 SKILL 定义中，可以提炼出下面的核心结构：

| 实体 | 含义 |
| --- | --- |
| `Symbol` | 抽象符号节点，带情绪向量、来源片段、载体信息 |
| `FantasyKnot` | 重复信息与幻想结，负责描述高频结构张力 |
| `IndexicalOperation` | 切片、凝固、对象化等范指动作 |
| `Proposition / Rule` | 命题规则，必须显式声明边界与失败条件 |
| `PowerChain` | 权力链运动，用于解释改写和规则迁移 |

这些结构由 skill 1 产出后，直接传递给 skill 2 进行形式化。

### 3.2 Skill 2 产出：可执行引擎模板

`object-engine-generator-v1.0` 将 skill 1 的符号系统整理为：

- `symbol_engine*.json`：完整的结构化引擎配置（符号、变量、规则、状态、任务、输出索引）
- `data_summary.md`：人类可读的引擎摘要
- 可视化图表（状态流、任务 DAG）
- 执行日志

### 3.3 Skill 3 产出：`game-engine-package` 压缩包

`game-engine-generator-v1.0` 将 skill 2 的输出与体验文档合并打包，最终产出包含以下内容的压缩包：

| 包内路径（相对路径） | 内容 | 来源 |
| --- | --- | --- |
| `data/symbol_engine_*.json` | 动态游戏规则 | skill 2 产出 |
| `data/data_summary.md` | 引擎配置摘要 | skill 2 产出 |
| `data/research_notes/*.md` | 公理、结构、规则补充、输出样本 | skill 1 产出 |
| `charts/state_flow.mmd` | 状态流转图 | skill 2 产出 |
| `logs/rule_engine_pseudo.txt` | 执行逻辑伪代码 | skill 2 产出 |
| `game-engine-generator-prompt.md` | 系统提示词 | skill 3 生成 |

### 3.4 运行时循环保留

外部规范把系统运转模式拆成了稳定阶段：

1. 摄入话语流与工具信息。
2. 提取重复模式。
3. 做对象化和符号化。
4. 生成幻想结、命题、边界和失败条件。
5. 做追问驱动与巫术风险检测。
6. 基于权力链和现实映射变量做规则改写。
7. 输出规则集、改写日志、沉默报告和追问种子。

这说明 skill 1 的产出不应只包含静态卡牌，还应该保留：

- 规则来源
- 改写痕迹
- 风险标记
- 可追问的问题种子

### 3.5 目标配置块

结合外部设计文档中的映射协议，`game-engine-package` 压缩包内的配置应至少覆盖：

| 配置块 | 内容 |
| --- | --- |
| `meta` | 来源名称、版本、领域、生成时间 |
| `symbolDictionary` | 符号表及其定义 |
| `variableModel` | 变量与现实映射权重 |
| `ruleCatalog` | 命题、触发条件、效果、边界、失败条件 |
| `runtimeRuleCards` | 运行时直接显示和执行的规则卡 |
| `stateModel` | 当前状态、历史、状态转移 |
| `promptAssets` | 系统提示词、约束、示例片段 |
| `resourceManifest` | 素材、图表、日志、研究笔记索引（相对路径） |
| `trace` | 源字段到目标字段的证据链 |
| `reviewQueue` | 无法确定或低置信字段 |

## 4. 和 `SymbolCatalyst` 当前资产的映射

`SymbolCatalyst` 不是空仓，它已经有一套资产可以支撑 skill 流水线的执行。按角色理解如下：

| 当前项目资产 | 在 skill 流水线中的角色 |
| --- | --- |
| `02_角色卡示例x8.md` | Skill 3 的输入：Persona/Card 结构样本 |
| `03_世界观骨架模板与示例.md` | Skill 3 的输入：世界观和设定模板 |
| `04_规则卡示例x12.md` | Skill 3 的输入：规则卡表达方式与运行时文案参考 |
| `05_叙事章节Demo与动态改写算法.md` | Skill 3 的输入：场景推进和规则改写逻辑样本 |
| `files/data/symbol_engine_20260218_v2.0.json` | Skill 2 的产出样本 / Skill 3 的输入主数据 |
| `files/research_notes/*` | Skill 1 的产出样本 / 语义补层 |
| `files/charts/state_flow.mmd` | Skill 2 的产出样本 / 状态流转关系 |
| `files/logs/rule_engine_pseudo.txt` | Skill 2 的产出样本 / 伪代码与执行参考 |
| `skills/symbol-engine-generator-v2.0/` | Skill 1 定义 |
| `skills/object-engine-generator-v1.0/` | Skill 2 定义 |
| `skills/game-engine-generator-v1.0/` | Skill 3 定义 |

因此，对 `SymbolCatalyst` **最准确的定位**是：

- 承载三个 skill 的 agent 工作空间
- Skill 流水线的执行环境
- `game-engine-package` 的生产车间

而不是：

- 只做资产归档的静态仓库
- 需要另建转换模块的中间层
- 直接面向最终玩家的运行时应用

## 5. 对当前项目的落地建议

### 5.1 本项目内应该继续完成什么

在 `SymbolCatalyst` 里，最值得推进的工作是：

- **将三个 skill 集成到 agent 配置中**，使其可以按流水线顺序自动执行。
- **完成 skill 3 的打包逻辑**，确保最终输出 `game-engine-package` 压缩包。
- **统一压缩包内的路径为相对路径**，保证可移植性。
- **补齐压缩包结构规范**，明确哪些文件必须包含、命名约定、目录布局。
- **用现有 `symbol_engine_20260218_v2.0.json` 做第一份端到端验证**，确认生成的包可被角色设定评价系统导入。

### 5.2 本项目内不必承担什么

当前项目没必要直接承担以下职责：

- 角色设定评价系统的运行时 UI
- 角色设定评价系统的导入解析逻辑（那是消费方的责任）
- 面向最终玩家的交互逻辑

### 5.3 最小下一步

如果只做最小推进，建议分三步：

1. **配置 agent**：将三个 skill 按顺序注册到 agent 中，定义流水线触发条件和数据传递协议。
2. **完善 skill 3 打包**：确保 `game-engine-generator-v1.0` 在打包时使用相对路径，产出符合导入协议的 zip。
3. **端到端验证**：用现有 `files/` 下的样本数据，走一遍完整的 skill 1 → 2 → 3 流水线，验证生成的包是否可以被角色设定评价系统导入。

## 6. 一句话结论

外部"角色设定评价系统设计师与引擎规范"描述的三个 skill 本身就是完整的转换流水线：把自然语言输入经过符号提取 → Schema 形式化 → 交付打包，最终输出可被角色设定评价系统直接导入的 `game-engine-package` 压缩包；`SymbolCatalyst` 的工作就是把这三个 skill 集成到 agent 中并确保打包产物使用相对路径、结构合规。
